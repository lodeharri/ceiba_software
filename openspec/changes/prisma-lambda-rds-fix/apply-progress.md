### Deviations from Design

- **`buildPrismaUrl` signature changed** from
  `buildPrismaUrl(rawUrl, stage, connectionLimit)` (3 args) → `buildPrismaUrl(rawUrl, stage)` (2 args).
  Design.md §3.15 preserved SSL semantics; design D-2 Correction establishes
  that the URL `?connection_limit` is ignored by `PrismaPg` regardless of
  input shape, so the `connectionLimit` parameter is dead data under the
  new adapter. This is a **narrow, intentional API contraction** — the
  only caller (`shared/prisma-client.ts` itself) is updated in the same
  change. No external callers exist (`grep -rn "buildPrismaUrl" packages/backend/src`
  returns only the factory and the new test file). Backward-compat for
  `getPrismaClient` is preserved via the `connectionLimit?` alias that
  translates to the internal `pg.Pool({ max })` (per user constraint).
- **Removed the eager `prisma.$connect()` call** from
  `getPrismaClient()`. PrismaPg owns pool lifecycle; calling `$connect()`
  synchronously after construction would defeat the lazy-first-query
  benefit of the adapter. Not a regression — the old call was redundant
  (Prisma auto-connects on first query) and arguably anti-pattern for cold
  starts.
- **5 BC `bootstrap.ts` files were NOT path-swapped**. They import
  `{ getPrismaClient, type PrismaLike }` from `../shared/prisma-client.js`,
  which re-exports `PrismaLike = PrismaClient` from the generated path. The
  indirection means zero bootstrap churn. The task 2.9 wording listing
  "5 bootstrap.ts" was over-cautious — verified by direct read of each
  file. The `orders/interface/handlers/bootstrap.ts` (the 6th bootstrap-style
  file) also does not need a path swap for the same reason.

### Deviations from Design

1. **Fix 2 wasm-engine hypothesis (user prompt)**: The prompt instructed installing `@prisma/schema-engine-wasm` + `@prisma/prisma-schema-wasm`. I tried this first (task 8.4–8.5), but it caused `prisma migrate deploy` to invoke the WASM engine whose RPC schema (`migrationsDirectoryPath`) differs from what the CLI sends (`migrationDirectories: []` array). I reverted this and shipped the native binary via `cp` instead. The WASM hypothesis was based on the assumption that prisma 6.x CLI requires the WASM packages at runtime — verified false: the CLI bundles `prisma_schema_build_bg.wasm` in its own `build/` directory and auto-selects it.

2. **Fix 2 binary-cp source path (self-correction)**: First attempt used `node_modules/.pnpm/@prisma+engines@6.19.3/...` (the pnpm store path). Tests failed because that path doesn't always exist on the host. Switched to `packages/infra/node_modules/@prisma/engines/` which is the workspace's stable, pnpm-curated location.

3. **Fix 2 bundling strategy**: Two strategies tried sequentially:
   - **Strategy A (REJECTED, bundle-size blocker)**: `PRISMA_CLI_BINARY_TARGETS=rhel-openssl-3.0.x` + run postinstall. The bundling container installed BOTH binaries (npm cache behavior — debian + rhel both copied), inflating the bundle to 97 MB zipped → 279 MB unzipped > 250 MB Lambda limit.
   - **Strategy B (CURRENT)**: `npm install --ignore-scripts` (no postinstall → no auto-fetched binary), then `cp` the rhel binary into the bundle in `afterBundling` from the workspace's pre-installed `packages/infra/node_modules/@prisma/engines/`. Bundle size well under limit; binary in correct path; but the CLI still uses WASM engine.

4. **Migrations directory copy**: The prompt did not mention explicitly that the `backend/prisma/migrations/` directory needs to be in the bundle, but the second deploy exposed this when the CLI read the schema's directory and got `migrationDirectories: []`. Added `cp -R ${PRISMA_MIGRATIONS_ABS}/. ${outputDir}/backend/prisma/migrations/` to `afterBundling`.

5. **`PRISMA_SCHEMA_ENGINE_BINARY` env var**: Added as a belt-and-suspenders measure (works on Prisma 5.x; harmless on Prisma 6.x where the CLI auto-selects WASM). Documented as a known limitation.
